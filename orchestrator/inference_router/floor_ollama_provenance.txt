What this file is

This file is the content-addressed declaration that the operator of
this Xion checkout has named the local Ollama daemon as their
Invariant-17 open-weights floor provider. A Witness who re-runs
`xion-verify inference-sovereignty` on a fresh clone recomputes the
sha256 of this file and matches it against the manifest pin. That
check proves the operator made the declaration and that the
declaration has not been silently rewritten on disk since.

What this file is NOT

This file does NOT content-address the model weights themselves. It
does not claim that the operator has pulled a specific byte-identical
GGUF blob. That stronger pin — the one Invariant 17 clause 2(iii)
envisions via the full-hash branch for upstream open-weights
artifacts — landed alongside this file in Phase 5g-viii as the
`gemma4-e4b-it-q4-k-m-gguf` `model-blob` entry in the same manifest.
This `provenance-record` entry stays in place because it pins a
distinct property (the operator's runtime declaration about which
daemon serves the floor); the two entries are companions, not
replacements. The strengthening history is tracked in
`KNOWN_WEAKNESSES.md` under `KW-INFERENCE-001`.

Why ship a provenance-record pin today

Invariant 17 requires a healthy `open_weights_self_hostable` floor
provider to be registered and listed in the manifest before the
Inference Router will bootstrap. Until the first real pin lands,
every live deployment's Router refuses to bootstrap off the pinned
sentinel alone (`sentinel-llm-v0` is a synthetic placeholder). The
provenance-record format is the smallest honest mechanism that:

  1. Adds a real, grep-able, content-addressed artifact to the
     manifest — not a silent edit, not an env-var back door, not a
     bypass flag.
  2. Preserves the verifier's structural guarantee: every floor entry
     has an on-disk file whose sha256 matches the pin.
  3. Names its own weakness in the machine-readable `format` field
     (`provenance-record`) so that future verifier extensions can
     distinguish provenance-records from real content-addressed
     model pins without reading prose.
  4. Leaves the path to the stronger pin explicit in
     `KNOWN_WEAKNESSES.md` § `KW-INFERENCE-001` rather than declaring
     victory prematurely. (Phase 5g-viii closes that residual by
     adding a sibling `model-blob` entry; this provenance-record
     entry remains the runtime-daemon-declaration anchor.)

Runtime configuration this declaration covers

At lifespan startup, the orchestrator registers
`OllamaGenerativeProvider` with `provider_id = "ollama"`. The Router's
`bootstrap()` check accepts a provider whose `provider_id` appears in
the manifest's `open_weights[]` list, whose `category` is
`open_weights_self_hostable`, and whose `health()` returns True. The
health check requires:

  - The Ollama daemon responds to `GET /api/tags` at
    `XION_OLLAMA_URL` (default `http://localhost:11434`).
  - The configured floor model (`XION_OLLAMA_FLOOR_MODEL`) is listed
    in that response — i.e. the operator has pulled it locally.

The configured floor model is operator-chosen, open-weights, and
locally health-checkable. Genesis Default is `gemma4:e4b-it-q4_K_M`
per `docs/26-INFERENCE-POLICY.md` § "The floor-model choice
(Gemma 4 E4B-it)" (rotated 2026-04-23 from the Phase-5g-i pin
`gemma3:4b` as part of Phase 5g-viii; rollback to `gemma3:4b` is one
env-var). The license posture strengthened in the same rotation:
`gemma3:4b`'s Custom Gemma TOU is replaced by Gemma 4's Apache 2.0
license, which satisfies Invariant 17 clause 2(i) without requiring
a Witness to accept any Google-specific terms.

Trust surface this declaration makes explicit

A Witness reading the manifest sees three entries as of Phase
5g-viii:

  - `sentinel-llm-v0` — the pre-genesis structural anchor. Proves the
    Router + verifier wiring works end-to-end. Kept in place.
  - `ollama` (this record) — the runtime-daemon-declaration anchor.
    Held by a runtime health check against a local daemon. The
    manifest pin commits to the declaration, not to the model blob.
  - `gemma4-e4b-it-q4-k-m-gguf` — the model-blob pin landed in Phase
    5g-viii. Content-addresses the upstream Hugging Face GGUF the
    operator should be running (the full-hash branch of Invariant 17
    clause 2(iii)). A Witness with internet access and a sha256
    implementation can independently re-verify the bytes; an operator
    who has not yet downloaded the artifact resolves to
    `NOT_YET_SEALED` for that entry only and the rest of the floor
    verifies normally.

A malicious operator can still swap the local Ollama daemon's
served weights without *this* file changing on disk. The
`gemma4-e4b-it-q4-k-m-gguf` model-blob entry closes that gap on the
Witness side (Witnesses can verify the bytes the operator should be
running); a runtime gate that refuses bootstrap on local-blob
mismatch remains a future hardening layer Phase 5g-viii deliberately
does not ship (it would couple Xion's floor to Ollama's internal
blob layout).

Operator identity statement

The operator running this checkout declares, by committing this file
to the repository and pinning its sha256 in
`open_weights_manifest.json`, that:

  - The Invariant-17 floor is held, at runtime, by the local Ollama
    daemon registered as `provider_id = "ollama"`.
  - The configured floor model is an open-weights model whose license
    permits local self-hosting.
  - The operator accepts responsibility for the integrity of the
    locally-pulled model bytes; the `gemma4-e4b-it-q4-k-m-gguf`
    model-blob entry in the same manifest gives a Witness the means
    to independently verify those bytes when the operator has set
    `XION_OPEN_WEIGHTS_GGUF_PATH` per the runbook in
    `docs/13-OPERATIONS.md` § "First-time GGUF setup".

This declaration is public, append-only (subsequent rotations commit
a new provenance record with its own sha256 pin, not a silent
rewrite), and falsifiable by any Witness who re-hashes the file.
